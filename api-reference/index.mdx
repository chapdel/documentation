# API Reference

Welcome to the Notch Pay API reference. This documentation provides detailed information about the endpoints, request parameters, and response formats for the Notch Pay API.

## Base URL

All API requests should be made to the following base URL:

```
https://api.notchpay.co
```

## Authentication

All requests to the Notch Pay API must be authenticated using your API key. Include your API key in the `Authorization` header of each request:

```
Authorization: YOUR_API_KEY
```

You can find your API keys in your [Notch Pay Business suite](https://business.notchpay.co) under Settings > API Keys.

## API Environments

Notch Pay provides two environments:

1. **Test Environment**: Use this environment for development and testing. Test API keys start with `test_`.
2. **Live Environment**: Use this environment for production. Live API keys start with `live_`.

Both environments use the same base URL, but the behavior differs based on the API key you use.

## Response Format

All API responses are returned in JSON format and follow a consistent structure:

```json
{
  "code": 200,
  "status": "OK",
  "message": "Descriptive message",
  "data": { ... }
}
```

## HTTP Status Codes

The API uses standard HTTP status codes to indicate the success or failure of a request:

| Code | Description |
|------|-------------|
| 200 | Success |
| 201 | Resource created |
| 202 | Accepted |
| 400 | Bad request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Not found |
| 422 | Unprocessable entity |
| 500 | Server error |

## Error Handling

When an error occurs, the API returns an appropriate HTTP status code and an error message:

```json
{
  "code": 400,
  "status": "Bad Request",
  "message": "Error message",
  "errors": {
    "field_name": ["Error description"]
  }
}
```

## Rate Limiting

The Notch Pay API implements rate limiting to protect against abuse. If you exceed the rate limit, you'll receive a `429 Too Many Requests` response.

## API Versioning

The current version of the API does not require a version specifier in the URL. Future versions may be accessed by including a version number in the URL or headers.
